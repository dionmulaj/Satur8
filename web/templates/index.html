Wiki — Detection Reference Satur8 v1.0
Detection Categories
DEAUTH_ATTACK Detects 802.11 deauthentication frame floods. An attacker sends forged deauth frames to forcibly disconnect legitimate clients from their access point — a prerequisite for many credential-capture and handshake-collection attacks.
ARP_SPOOFING Identifies ARP cache poisoning where two different MAC addresses claim ownership of the same IP address. Classic MITM setup — the attacker inserts themselves between a victim and the gateway to intercept or modify traffic.
ARP_FLOOD Raised when a single MAC address sends an abnormally high volume of ARP requests within a short window. May indicate network reconnaissance, a misconfigured device, or a denial-of-service tool. A one-time-window grace period on startup prevents false positives from interface buffer flushes.
EVIL_TWIN Detects rogue access points advertising a known SSID from a different BSSID (MAC address). Attackers use evil twins to lure clients into connecting to a controlled AP where all traffic can be intercepted. A baseline scan establishes trusted SSID→BSSID mappings before alerting.
KARMA_ATTACK Flags an AP that responds to multiple distinct probe requests with matching beacons, impersonating whatever SSID a client is looking for. Used by rogue APs (e.g. WiFi Pineapple) to automatically trap devices that broadcast previously joined networks.
ROGUE_BEACON An access point is broadcasting beacons for an SSID that was never seen during the baseline environment scan. May indicate a newly deployed rogue AP attempting to blend in with the environment after the baseline was collected.
PINEAPPLE_MULTI_SSID A single BSSID is broadcasting beacons for an unusually large number of distinct SSIDs simultaneously — a behavior characteristic of WiFi Pineapple devices operating in PineAP/broadcast mode.
SUSPICIOUS_DEVICE A discovered client's MAC OUI matches a known offensive hardware vendor (e.g. Hak5 / WiFi Pineapple families). The OUI database is loaded from oui.txt at startup.
SUSPICIOUS_SSID Beacon or probe contains an SSID matching known attack tool defaults or suspicious patterns (e.g. "Pineapple", "Evil Twin", default Karma SSID names).
BASELINE_COMPLETE Informational notice emitted when the baseline environment scan finishes. The collected SSID→BSSID mappings are saved to disk and used as the trusted reference for subsequent Evil Twin and Rogue Beacon detection.
SCAN_STARTED Emitted each time packet capture begins. Records the capture interface, current channel, platform, and whether channel hopping is active. Useful for auditing when monitoring sessions were started.
SCAN_STOPPED Emitted when monitoring is stopped. Includes session duration and total packets captured — provides a quick summary of each monitoring session.
MONITOR_MODE_FALLBACK Fired when the capture interface cannot be opened in monitor mode and Satur8 falls back to standard promiscuous capture. In this degraded state, 802.11 management frames (Beacons, Deauths, Probe Requests) will not be visible, disabling most wireless attack detection. Action: disconnect from WiFi and run with sudo.
OUI_DB_LOADED Confirms the OUI vendor database loaded successfully at scan start. Reports the number of entries available for vendor identification and the file path used.
OUI_DB_MISSING The oui.txt database was not found in the project root. Vendor names will show as "Unknown" and OUI-based suspicious device detection will be unavailable. Place oui.txt in the project root and restart.
How It Works
Packet Capture Satur8 uses Scapy to sniff raw 802.11 and ARP frames from a wireless interface placed in monitor mode. On macOS, libpcap via conf.use_pcap = True is used; on Linux, standard raw socket capture applies.
Baseline Scan Before threat detection begins, an optional baseline scan (default 300 s) observes the local RF environment and builds a map of trusted SSID→BSSID associations. This reduces false positives for Evil Twin detection in dense Wi-Fi areas. The baseline is persisted to baseline.json and reloaded on each run.
Thresholds Detection thresholds are configurable via environment variables: ARP_THRESHOLD (default 200 req/10 s), DEAUTH_THRESHOLD (default 15 frames/10 s). All settings live in config.py.
OUI Lookup A local oui.txt database maps MAC prefixes to vendor names, enabling vendor-based device identification and suspicious-hardware flagging without any external API calls.
Real-time UI Alerts, packet feeds, and device tables are pushed to the browser over WebSocket (Socket.IO) so the dashboard updates live without polling.
